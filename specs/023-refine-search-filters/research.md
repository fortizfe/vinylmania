# Phase 0 Research: Refine Search Filters Usability

All items in the Technical Context were resolved directly from the existing codebase (no `NEEDS CLARIFICATION` markers remained), so this research consolidates the concrete decisions and their rationale rather than resolving open unknowns.

## 1. Icon implementation for Apply/Clear (FR-010–FR-013)

- **Decision**: Implement the Apply/Clear icons as hand-written inline SVG components (one `ApplyIcon`, one `ClearIcon`), rendered inside the existing `Button` component's `size="icon"` variant, with an `aria-label` on each `<button>` (or a visually-hidden `sr-only` label) to satisfy FR-013.
- **Rationale**: The project has no icon library installed (`lucide-react`, `react-icons`, `heroicons`, etc. are all absent from `frontend/package.json`). The existing convention for icons (`HeaderNavIcons.tsx`, `HamburgerMenu.tsx`) is hand-written `<svg>` elements using `viewBox="0 0 20 20" fill="none" stroke="currentColor" strokeWidth={2}` with Tailwind sizing classes (`h-4 w-4`). `Button` already exposes a ready-made `size="icon"` variant (`iconButtonClassName`, 36×36px hit target) used elsewhere. Matching this convention satisfies Principle III (Simplicity/YAGNI — no new dependency for two icons) and keeps visual consistency with the rest of the header/nav UI.
- **Alternatives considered**: Adding a small icon library (e.g., `lucide-react`) — rejected as an unjustified new dependency for two icons when an established in-repo pattern already exists (Principle III). Using text-based symbol characters (e.g., "✓" / "✕") instead of SVG — rejected because it doesn't match the project's existing icon convention and has weaker cross-platform/font rendering guarantees than SVG.

## 2. Live format-label text vs. abbreviation (FR-002–FR-007)

- **Decision** (as implemented — supersedes the pixel-measurement approach originally proposed here): Compute the label from the component's local, not-yet-applied `selectedFormats` state (owned by the new `FormatFilter`), in the order values were toggled on. Fit is determined by comparing the full comma-joined string's character length against a fixed budget (`MAX_LABEL_LENGTH = 24` in `FormatFilter.tsx`) rather than measuring real rendered pixel width. Falls back to `"{first} (+{selectedFormats.length - 1})"` when the joined string exceeds that length.
- **Rationale**: `selectedFormats` already exists as in-memory state in the current `SearchFiltersControl.tsx` (`toggleFormat` appends in selection order via `[...prev, option]`), independent of the `Apply filters` submit handler — this is exactly the "in-progress, not-yet-applied" state the spec's Assumptions section calls for. The project's unit test stack (Vitest + jsdom, see `vitest.config.ts`) has no real layout engine — `clientWidth`/`scrollWidth`/`getBoundingClientRect()` are always `0` in jsdom, and `ResizeObserver` isn't polyfilled — so a `ResizeObserver`-driven pixel-measurement approach would be effectively untestable without heavy per-element mocking. Since the spec's Assumptions section explicitly leaves the exact threshold as an implementation detail, a character-length heuristic satisfies FR-006 deterministically and is trivially unit-testable (as done in `FormatFilter.test.tsx`), at the cost of being an approximation of true available width rather than an exact fit.
- **Alternatives considered**: Real pixel measurement via a hidden offscreen span + `ResizeObserver` (the approach originally recorded here) — rejected during implementation once it became clear it cannot be exercised by the project's jsdom-based unit tests without mocking every relevant DOM measurement API per test case, which is significantly more complex than the behavior it verifies. Deferring the abbreviation entirely to CSS `text-overflow: ellipsis` — rejected because it doesn't satisfy FR-006's explicit "first value + count" content requirement (ellipsis alone loses the count).

## 3. Component decomposition for library-first extensibility (FR-015–FR-016)

- **Decision**: Split `SearchFiltersControl.tsx` into: `FormatFilter` (trigger button + live label + its own selection `Modal`), a generic `TextFilterField` (compact free-text input, parameterized by id/label/value/onChange, reused for both Genre and Style), and `FilterActions` (the icon-only Apply/Clear group). `SearchFiltersControl` itself becomes a thin container that holds shared form state/handlers and renders the three pieces in Format-first order (FR-001).
- **Rationale**: This directly satisfies FR-015 (each filter is an independent, reusable component) and FR-016 (a future filter, e.g. "Year", would be added as a new sibling component plus one line in the composing container, without touching `FormatFilter`, `TextFilterField`, or `FilterActions` internals). It also aligns with Principle IV (SOLID: Single Responsibility) and the existing project convention of small, focused components under `components/ui/` and `components/`.
- **Alternatives considered**: Keeping one large `SearchFiltersControl` component with conditional rendering per filter type — rejected as directly contradicting the explicit clarification (FR-015/016) and Principle IV. Introducing a generic config-driven "filter registry" (e.g., an array of filter definitions rendered dynamically) — rejected as premature abstraction (Principle III/YAGNI): only one additional filter type (Format) differs structurally from the two free-text ones today, so a declarative registry would add indirection without a second concrete use case yet to justify it.

## 4. Testing approach

- **Decision**: Unit-test each new component in isolation with Vitest + React Testing Library (label text for each selection-count scenario, compact sizing class presence for `TextFilterField`, icon-only rendering + accessible name for `FilterActions`), and extend the existing Playwright spec (`e2e/tests/search-result-filters.spec.ts`) to cover the Format-first DOM order, a live label update without clicking Apply, and that Apply/Clear remain operable via their accessible names.
- **Rationale**: Matches the project's existing test layout (`frontend/tests/unit/*.test.tsx`, one Playwright spec per feature area) and the constitution's Test-First principle (failing tests precede implementation) plus its mandatory e2e coverage gate for `/frontend` PRs.
- **Alternatives considered**: Relying solely on the existing single `SearchFiltersControl.test.tsx` without per-component tests — rejected because it would not independently verify each new reusable component's contract, undermining the "independently testable" intent behind FR-015.

## 5. Versioning & changelog

- **Decision**: Bump `frontend/package.json` from `0.13.0` to `0.14.0` (MINOR) and add a corresponding `## [0.14.0]` entry to `frontend/CHANGELOG.md` under `Changed` (filter bar layout/behavior) in the same PR.
- **Rationale**: Constitution Principle VI classifies additive, backward-compatible UI behavior changes as MINOR; the Development Workflow gate requires the changelog entry and version bump to land together, with no `[Unreleased]` staging section (continuous deployment on merge to `main`).
- **Alternatives considered**: PATCH bump — rejected because this introduces new user-visible behavior (live label, icon-only actions, reordered layout), not just a fix/clarification.
