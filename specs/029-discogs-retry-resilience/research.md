# Phase 0 Research: Discogs Communication Resilience (Retry Policy)

All items below were resolved from the existing codebase (`backend/src/discogs/discogsClient.ts`, `discogsErrors.ts`, `backend/src/cache/cacheAside.ts`, `backend/src/library/libraryEnrichment.ts`, `backend/src/discogs/collection/collectionClient.ts`) plus the spec's Clarifications session (2026-07-10). No unresolved `NEEDS CLARIFICATION` markers remain in the Technical Context.

## 1. Where retry logic lives: the shared axios interceptor, not per call site

- **Decision**: Implement retry-with-backoff inside the existing response error interceptor of `createDiscogsHttpClient()` in `discogsClient.ts`, before it maps a failure to a typed `DiscogsError`. On a retryable failure, the interceptor re-issues the original request (tracking an attempt counter on the axios request config, the same pattern used by `axios-retry` internally) instead of rejecting immediately; only once attempts are exhausted does it fall through to today's existing status-code-to-error mapping.
- **Rationale**: `getDiscogsHttpClient()` is the single choke point every scoped read already shares — `searchCatalog`, `getRelease`, `getMasterRelease`, `getMasterReleaseVersions`, `getArtist` (all in `discogsClient.ts`) and, transitively, `libraryEnrichment.ts`'s `enrichEntry` (which calls `getRelease`). Adding retry once here satisfies FR-009's clarified scope (interactive catalog reads **and** the background library-enrichment path) with zero changes to any call site — the same reasoning that made `withCache` wrapping effective for caching (feature 011) applies here.
- **Alternatives considered**: Wrapping each of the five/six exported functions individually with a `retryable(fn)` helper (rejected — duplicates the same wrapping six times for no benefit, and risks the six call sites drifting out of sync over time, violating DRY/Simplicity); a dedicated retrying HTTP client swapped in only for some functions (rejected — reintroduces the exact scope-inconsistency the clarification session explicitly resolved against).

## 2. Retry mechanism: hand-rolled, no new dependency

- **Decision**: Add a small, dependency-free module `backend/src/discogs/discogsRetry.ts` exporting a pure `isRetryable(error)` classifier and a `backoffDelayMs(attempt)` schedule function, consumed by the interceptor. No retry library (`axios-retry`, `p-retry`, `cockatiel`) is added.
- **Rationale**: The project already prefers small, fully-owned infrastructure modules over pulling in a library for a bounded, well-understood problem — `backend/src/cache/cacheAside.ts` (feature 011) is the direct precedent: a hand-rolled ~80-line cache-aside wrapper instead of a caching library. A correct retry classifier + fixed backoff schedule is comparably small (~30 lines) and easier to unit test deterministically (Jest fake timers) and to keep in lockstep with this project's own `DiscogsError` subtypes than a generic library would be. Constitution Principle III requires the simpler alternative be used unless a documented reason justifies added complexity; no such reason exists here.
- **Alternatives considered**: `axios-retry` (rejected — external dependency for ~15 lines of retry-loop logic we can own and test directly against our own error types); `cockatiel` (offers retry + circuit breaker together, considered more seriously since both are needed here — still rejected because our circuit breaker only needs two states plus a cooldown timer (§4), well within what's comfortable to hand-roll and keep dependency-free, consistent with the `cacheAside.ts` precedent, and it keeps the whole feature index/import-free of new production dependencies).

## 3. Retryable-failure classification and backoff schedule

- **Decision**: A failure is retryable only when it is a 429 (rate-limited/busy) or a 5xx/network failure (today's `DiscogsUnavailableError` condition) — never a 404 (not found), never an auth failure (see §5), and never a validation/mapping error. Backoff uses one fixed schedule regardless of which retryable condition occurred (per Clarifications): attempt delays of `300ms`, then `900ms` (±20% jitter to avoid synchronized retry storms across concurrent requests), for at most 2 retries (3 attempts total) — combined backoff sleep ≈1.2s, safely inside the ~5 second budget in FR-010.
- **Rationale**: Directly implements FR-001–FR-004 and FR-010 as clarified (uniform schedule, bounded attempts/time). Jitter is a minimal, standard addition (not a new failure-type branch) that keeps many concurrent collectors' retries from re-hitting Discogs in lockstep, supporting FR-011/SC-006 without adding a new axis of configuration.
- **Alternatives considered**: Reading Discogs' `Retry-After`/`x-discogs-ratelimit-*` headers to size the wait (rejected by the clarification session in favor of one fixed schedule — Discogs' `/database/search` and catalog endpoints do not reliably send `Retry-After` on 429 today, so this would add branching complexity for a signal that isn't consistently available).

## 4. Per-attempt HTTP timeout for retryable calls

- **Decision**: Reduce the per-attempt timeout used for the five retry-scoped calls (search, release, master, master versions, artist) from the client's current blanket `10_000`ms to `4_000`ms per attempt, applied via the existing per-request `timeout` option (the same mechanism already used for the excluded rating lookup's 2s timeout).
- **Rationale**: With up to 3 attempts, keeping the existing 10s timeout could compound to a ~30s worst case if Discogs merely hangs rather than errors quickly — directly undermining SC-002's "no perceptible hang." Three attempts at 4s (12s) plus ≈1.2s of backoff sleep keeps the worst-case bound well inside a reasonable page-load ceiling. This is a request-level tuning change with no effect on the cache TTL constants or the rating lookup's independent 2s budget.
- **Alternatives considered**: Leaving the 10s timeout unchanged (rejected — worst-case 30+s directly conflicts with SC-002); making the timeout configurable per environment (rejected — no requirement calls for that, YAGNI).

## 5. Auth-failure classification gap: 401/403 must map to `DiscogsAuthError` in `discogsClient.ts`

- **Decision**: Add explicit `401`/`403` handling to `createDiscogsHttpClient()`'s interceptor, mapping to the existing `DiscogsAuthError` (imported, not newly defined) — mirroring the pattern already implemented in `backend/src/discogs/collection/collectionClient.ts` (`status === 401 || status === 403 → DiscogsAuthError`). Today, `discogsClient.ts` has no such branch: a 401/403 currently falls through to the generic `DiscogsUnavailableError` case.
- **Rationale**: FR-002 requires the system to **never** retry when "the catalog service rejected the linked credentials." Without this fix, an auth failure on the catalog client (e.g. a misconfigured/expired `DISCOGS_TOKEN`) would be indistinguishable from a transient outage and would incorrectly consume retry attempts and trip the circuit breaker on every request until the token is fixed — the opposite of the intended behavior. This is a necessary correctness fix uncovered by implementing FR-002, not scope creep: the retry feature cannot correctly exclude auth failures without it.
- **Alternatives considered**: Leaving 401/403 unclassified and accepting they'd be retried like any other unavailable failure (rejected — directly violates FR-002, and would waste the entire retry budget plus falsely trip the circuit breaker during what is actually a static misconfiguration, not a transient outage).

## 6. Circuit breaker: two-state, in-memory, app-wide, hand-rolled

- **Decision**: Add `backend/src/discogs/discogsCircuitBreaker.ts`, a module-level singleton (same reuse pattern as `sharedClient` in `discogsClient.ts` and the Redis client in `cacheAside.ts`) implementing:
  - **CLOSED** (default): requests proceed normally through retry logic. Each request whose retries are *fully exhausted* (not each individual attempt) counts as one strike against a rolling window.
  - **Trip condition**: 5 strikes within a 30-second rolling window → **OPEN**.
  - **OPEN**: new requests fail immediately with the existing `DiscogsUnavailableError`, without making any network call or spending retry attempts, for a 20-second cooldown.
  - **HALF-OPEN** (after cooldown): the next single request is let through normally (its own full retry budget applies); success closes the breaker (strike count resets), failure reopens it and restarts the 20-second cooldown.
- **Rationale**: Implements FR-011/SC-006 as clarified. Counting only fully-exhausted requests (not every sub-attempt) means the breaker only reacts to failures retries could not paper over — a real signal of sustained trouble — rather than tripping on the same blips retries already handle. A rolling-window strike counter plus a single cooldown timer is the minimum state needed for this behavior and is small enough to unit-test deterministically with Jest fake timers, consistent with the "hand-rolled over library" decision in §2.
- **Alternatives considered**: A full-featured breaker library (`opossum`, `cockatiel`) — rejected per §2's reasoning; a stricter trip condition counting every attempt (rejected — would trip far too eagerly, since 3 attempts per request already means multiple "failures" per single ultimately-successful request); a Redis-backed shared breaker state across all serverless instances (rejected — see §7, adds real complexity for a benefit YAGNI doesn't justify at this scope).

## 7. Circuit breaker scope limitation: per-warm-instance, not globally coordinated

- **Decision**: Circuit breaker state lives in process memory only, exactly like the existing `sharedClient` and `inFlight` singletons. On the deployed Vercel serverless target, this means the breaker's protection is best-effort per warm container, not a single global state shared across every concurrent serverless instance.
- **Rationale**: This is an accepted, documented trade-off, not an oversight. A globally coordinated breaker would require a shared store (e.g. Redis-backed counters) and materially more complexity (races, TTL bookkeeping, cross-instance consistency) for a feature whose spec (FR-011) only asks that retries not "further amplify load" during an outage — even per-instance protection meaningfully reduces total retry volume without that added complexity, and nothing in the spec's Success Criteria (SC-006) requires exact global coordination.
- **Alternatives considered**: Redis-backed shared breaker counters (rejected for this increment per the above; could be revisited later if production data shows per-instance breakers are insufficient — flagged here for future reference, not turned into a spec requirement).

## 8. Scope boundaries confirmed out of this feature

- **Decision**: The community-rating enrichment lookup (`getReleaseRating`, its own 2-second timeout in `discogsClient.ts`) is untouched — no retry, no circuit-breaker interaction — per the spec's Assumptions. The OAuth-authenticated collection sync path (`collectionClient.ts` / `oauthHttpClient.ts`, used by `librarySyncService.ts` for per-user collection reads/writes) is a **separate axios instance** entirely and is out of scope: the Clarifications session's "background library-sync/enrichment operation" refers specifically to `libraryEnrichment.ts` (which calls `getRelease` — a scoped catalog read), not `librarySyncService.ts` (which performs per-user collection writes over a different client with its own existing 401/403/429 handling already in place).
- **Rationale**: Keeps the retry/circuit-breaker change confined to the single shared catalog HTTP client, exactly matching what the spec scoped in and out. `collectionClient.ts` already has its own complete error classification (including the 401/403 mapping §5 is adding to `discogsClient.ts`) and is unaffected.
- **Alternatives considered**: Extending retry to `collectionClient.ts`/`librarySyncService.ts` too (rejected — outside the spec's clarified scope; those are per-user write operations, not the "catalog reads" the clarification session referred to, and retrying writes safely requires idempotency analysis this spec never asked for).

## 9. Observability

- **Decision**: Extend `LogOutcome` in `backend/src/config/logger.ts` with one new value, `'circuit_open'` (logged at `warn` level when a request fails fast without attempting the network call). Existing `success`/`rate_limited`/`unavailable` log lines gain a `meta.attempts` field (integer, defaults to 1) so a log line where `attempts > 1` and outcome is `success` means "recovered after retry," and `attempts === 3` with outcome `rate_limited`/`unavailable` means "failed after exhausting retries" — satisfying FR-008/SC-005 without inventing a combinatorial set of new outcome strings.
- **Rationale**: Reuses the project's existing structured-logging pattern (Principle V) exactly as feature 011 did for cache hit/miss — a `meta` field addition is the smallest change that makes User Story 3's distinctions ("succeeded first try" vs. "succeeded after retry" vs. "failed after exhaustion") mechanically greppable.
- **Alternatives considered**: A dedicated metrics/APM pipeline (rejected — no existing metrics infra in this project, structured logs are the established pattern, consistent with feature 011's same conclusion).

## 10. Testing strategy

- **Decision**: Unit tests for `discogsRetry.ts` (classifier + backoff schedule, pure functions, no timers needed) and `discogsCircuitBreaker.ts` (state transitions, using Jest fake timers to control the rolling window and cooldown deterministically — a technique not yet used elsewhere in this backend test suite, introduced here for the first time). Contract tests in `backend/tests/contract/discogsClient.contract.test.ts` extended with `nock` scenarios: fail-once-then-succeed (retry recovers), fail-every-attempt (exhaustion), 404/401/403 (no retry), and a circuit-open scenario. A new integration test exercises the actual routes end-to-end (`backend/tests/integration/discogsRetryResilience.test.ts`) covering US1–US3.
- **Rationale**: Matches Principle I (Test-First) — every new behavior branch (retryable vs. not, exhausted vs. recovered, breaker open/half-open/closed) needs a failing test before implementation, consistent with how feature 011 introduced `ioredis-mock` for its own new infrastructure.
- **Alternatives considered**: Using real `setTimeout` delays in tests (rejected — would make the suite slow and flaky; fake timers are the standard Jest approach for deterministic backoff/cooldown testing).
