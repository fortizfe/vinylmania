# Quickstart: Validating Discogs Communication Resilience

Prerequisites: this feature is implemented (see `tasks.md` once generated by `/speckit-tasks`). This guide validates the retry policy and circuit breaker end-to-end; it does not include implementation code.

## Prerequisites

- `backend/.env` (or shell env) with the usual `DISCOGS_TOKEN`, `FIREBASE_*` vars.
- Backend dev server running: `npm run dev` in `backend/`.
- A way to make Discogs fail transiently on demand. The automated test suite (below) uses `nock` for this; for manual verification, point `DISCOGS_BASE_URL` at a small local stub server you control (e.g. a throwaway Express app) that can be scripted to return 429/500 for the first N requests to a given path and then behave normally.

## 1. Automated checks (primary validation)

- `npm test` in `backend/` — the full suite must pass, including:
  - `backend/tests/unit/discogsRetry.test.ts` and `backend/tests/unit/discogsCircuitBreaker.test.ts` (new)
  - `backend/tests/contract/discogsClient.contract.test.ts` (extended: retry-then-succeed, retry-exhausted, no-retry-on-404/401/403, circuit-open scenarios)
  - `backend/tests/integration/discogsRetryResilience.test.ts` (new — route-level, covers US1/US2/US3)
- These are the authoritative proof this feature works — the scenarios below are for manual/production spot-checks only.

## 2. Manual: master detail recovers from a transient "busy" response (User Story 1)

1. Configure the local stub (or a `nock`-style interceptor if testing against a script) so a request to `/masters/:id/versions` returns HTTP 429 once, then 200 on the next attempt.
2. Sign in to the app, search for any master release, and open its detail page.
3. **Expected**: the page loads normally with the version list populated — no "catalog service busy" message is shown. In the backend console, look for a JSON log line with `"route":"/masters/:id/versions"` (or similar), `"outcome":"success"`, and `"meta":{"attempts":2}` — confirming the request recovered after exactly one retry.

## 3. Manual: retries exhaust gracefully (User Story 1, edge case)

1. Configure the stub to return 500 on every attempt for a given master ID.
2. Open that master's detail page.
3. **Expected**: after a short, bounded delay (not a long hang), the collector sees the existing "temporarily unavailable, please try again" message. The backend log shows `"outcome":"unavailable"` with `"meta":{"attempts":3}`.

## 4. Manual: no retry on a genuinely missing release (edge case)

1. Request a release/master ID that Discogs returns 404 for.
2. **Expected**: the "not found" response returns immediately (no added delay from retries). The backend log shows `"outcome":"not_found"` with no `attempts` field elevated above 1 (first-attempt failure, correctly not retried).

## 5. Manual: circuit breaker trips under sustained failures (User Story with FR-011)

1. Configure the stub to fail every request across several different master/release/search IDs (simulating a broad outage, not a single retried request).
2. Issue 5+ distinct catalog requests in quick succession (different IDs, so none are coalesced by the existing single-flight cache layer).
3. **Expected**: after the 5th exhausted failure within the rolling window, subsequent requests fail immediately (no outbound HTTP call, no retry delay) — backend log shows `"outcome":"circuit_open"`. After ~20 seconds, the next request is let through as a trial; if the stub is reconfigured to succeed by then, subsequent requests resume normally.

## 6. Manual: background library enrichment benefits too (FR-009 scope)

1. With a Discogs-linked account and at least one library entry, configure the stub so the release lookup for that entry's `discogsReleaseId` fails once then succeeds.
2. Load the library list page (triggers `libraryEnrichment.enrichEntries`).
3. **Expected**: the entry renders with full catalog data (`catalogStatus: 'ok'`), not the degraded `catalogStatus: 'unavailable'` placeholder — confirming the shared retry logic applied to this background path too, without any change to `libraryEnrichment.ts` itself.
