# Feature Specification: Discogs Rate Limit Smoothing & Call Reduction

**Feature Branch**: `040-discogs-rate-limit-smoothing`

**Created**: 2026-07-13

**Status**: Draft

**Input**: User description: "Trabaja con la especificación detallada en @.hu/discogs-rate-limit-smoothing.md. Que no se lancen test e2e durante el proceso de implementación ya que hay que solucionar un bug primero"

## Clarifications

### Session 2026-07-13

- Q: Should Story 3's retry extension apply to non-idempotent collection writes (e.g. adding a release instance), which risk creating duplicate collection entries if retried after an ambiguous failure? → A: Retry only safely-idempotent collection calls (reads, `setRating`, `setFieldValue`, `deleteInstance`); exclude `addReleaseToCollection` from automatic retry — the collector retries that action manually if it fails.
- Q: Does the local throttle delay a live, user-facing request (search, detail page) synchronously, and if so, is there a maximum wait? → A: Yes, the delay applies synchronously within the request but is capped at a short maximum wait; once that cap is reached, the request is sent anyway rather than continuing to wait.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - The collector never notices the app throttling itself against Discogs (Priority: P1)

As a collector using Vinylmania (searching, viewing a release/master detail page, or
syncing a Discogs-connected library), I want the app to proactively stay under the
per-minute request budget Discogs grants it, so that my experience stays smooth and I
stop seeing the "service busy" message as often — instead of the app firing requests
at full speed and only reacting after Discogs has already rejected one with a 429.

**Why this priority**: This is the structural fix. Discogs' own documentation
explicitly asks integrators to "throttle its requests locally" instead of relying on
429 responses as the only signal. Today Vinylmania reads the rate-limit headers
Discogs returns on every response but never acts on them — this story closes that gap
and applies to every flow that talks to Discogs (search, detail pages, library sync).

**Independent Test**: Can be fully tested by driving a burst of catalog requests (e.g.
a page of search results or a full library sync) against a fresh process and observing
that outgoing Discogs requests get spaced out as the known remaining budget shrinks,
without any request being dropped or failing due to the throttle itself.

**Acceptance Scenarios**:

1. **Given** the client already knows, from the last Discogs response, the value of
   `X-Discogs-Ratelimit-Remaining`, **When** the number of requests remaining in the
   current window drops below a safety threshold, **Then** subsequent outgoing
   requests to Discogs are spaced out (queued/delayed) locally instead of firing
   immediately, so the window isn't exhausted and a 429 is avoided.
2. **Given** a burst of Discogs requests generated by a single collector action (e.g.
   several search results enriching at once, or a full library sync), **When** that
   burst executes, **Then** the actual rate of requests sent to Discogs respects the
   known limit (60/min authenticated, 25/min unauthenticated) without depending on a
   429 to slow it down after the fact.
3. **Given** the local throttle has spaced out a request, **When** the response
   reaches the collector, **Then** the added delay is the minimum necessary to avoid
   exceeding the limit — there is no fixed arbitrary wait burning time when the
   remaining budget is ample — and the wait is capped at a short maximum: if that cap
   is reached, the request is sent anyway rather than continuing to hold it, so a
   live, user-facing request (search, detail page) is never held indefinitely.
4. **Given** the OAuth collection client and the catalog client are two separate HTTP
   client instances calling the same Discogs server (same per-IP limit), **When** the
   local throttle is applied, **Then** both clients share the same request-per-minute
   budget/counter, not two independent counters that combined could still exceed
   Discogs' real limit.
5. **Given** the local throttle stays active, **When** Discogs still returns a 429
   anyway (because of other processes, another instance, or a limit change without
   notice, as the documentation itself warns), **Then** the existing retry policy and
   circuit breaker (already implemented) continue to apply unchanged as a safety net —
   this story adds a preventive layer, it does not replace the reactive one.
6. **Given** the OAuth collection client today never retries a 429, **When** this
   story is complete, **Then** that client is covered by the same preventive local
   throttle protection as the catalog client.

### User Story 2 - A single search never fires an uncontrolled burst of rating calls (Priority: P2)

As a collector searching the catalog, I want seeing search results (with their
community ratings) to not fire one Discogs call per result all at once, so that a
single search doesn't by itself consume a large share (or all) of the available
per-minute request budget.

**Why this priority**: This is the clearest concrete "reduce calls" finding: with the
default page size, a single cold-cache search can trigger dozens of simultaneous
rating calls, which alone can exhaust the per-minute budget. It's ranked below Story 1
because the preventive throttle protects more generally regardless of where the burst
comes from, but this story removes the single largest known source of bursts.

**Independent Test**: Can be fully tested by running a cold-cache search and observing
that the number of Discogs calls in flight at the same time for that search stays
bounded to an agreed concurrency limit, while the search still completes in a time the
collector perceives as fast.

**Acceptance Scenarios**:

1. **Given** a cold-cache search whose page results are mostly of a type eligible for
   rating enrichment, **When** enrichment runs, **Then** the number of Discogs calls in
   flight at the same time for that search is bounded to a reasonable concurrency
   limit (consistent with the concurrency limit already used for library enrichment,
   or the value determined during planning), instead of all firing at once.
2. **Given** that concurrency limit is applied, **When** the total search response time
   is compared against today's cold-cache behavior, **Then** the added time is
   acceptable to the collector (the search still feels fast) — the goal is to stop
   saturating the rate limit without making search perceptibly slow.
3. **Given** a search whose results are already cached (rating and/or master data
   within their existing TTLs), **When** that search is served, **Then** it is
   unaffected by the new concurrency limit, because no real Discogs call is triggered.
4. **Given** the existing fail-soft behavior of rating enrichment (a rating that fails
   or is slow is skipped instead of breaking the search), **When** the concurrency
   limit is applied, **Then** that fail-soft behavior is preserved exactly — this story
   only changes how many calls run in parallel, not what happens when one fails.

### User Story 3 - The collection client gets the same resilience as the catalog client (Priority: P3)

As the project owner, I want library sync and collection mutations (add/remove a
release, rate it, edit a field) to use the same retry-with-backoff policy that already
protects catalog calls, so that a transient 429 during a sync doesn't fail immediately
when the same kind of failure is already absorbed automatically for catalog calls.

**Why this priority**: This is a real inconsistency, but it's not the flow reported as
painful today, so it comes after the preventive throttle (Story 1) and the search
burst fix (Story 2).

**Independent Test**: Can be fully tested by simulating a transient 429/5xx on a
collection call (e.g. a page of a library sync, or a rating mutation) and observing
that it retries with backoff instead of failing on the first attempt.

**Acceptance Scenarios**:

1. **Given** a safely-retryable collection call (a page read during a full library
   sync, `getFieldMap`, or an idempotent mutation such as `setRating`,
   `setFieldValue`, or `deleteInstance`) fails with a 429 or a transient
   5xx/network error, **When** that failure occurs, **Then** the collection client
   retries with the same growing-backoff criteria and attempt limit already defined
   for the catalog client, instead of failing on the first attempt.
2. **Given** a non-transient failure in the collection flow (revoked credentials,
   not-found), **When** it occurs, **Then** the collection client still does not retry
   and translates it to the corresponding error exactly as it does today — this
   behavior does not change.
3. **Given** the existing app-wide circuit breaker, **When** retry behavior is extended
   to the collection client, **Then** both the catalog and collection clients
   contribute to the same exhausted-failure counter and share the same circuit state,
   since both consume the same per-IP Discogs limit.
4. **Given** `addReleaseToCollection` is a non-idempotent write (retrying it after an
   ambiguous failure risks adding the same release to the collector's library twice),
   **When** it fails with a 429 or a transient 5xx/network error, **Then** the
   collection client does NOT automatically retry it — the failure is surfaced to the
   collector exactly as it is today, so they can choose to retry the action themselves.

### Edge Cases

- What happens if the shared store backing the request counter (if one is used) is
  unavailable? The throttle must degrade fail-soft — it must never block a legitimate
  request indefinitely nor drop a request because the throttle mechanism itself failed.
- What happens when the backend process starts cold and hasn't yet received any
  response from Discogs (doesn't know `X-Discogs-Ratelimit-Remaining` yet)? It must
  assume a conservative default budget (based on the documented authenticated limit)
  until it receives the first real response.
- What happens in a deployment with several backend instances running in parallel
  (e.g. serverless)? A throttle that only tracks state within a single process cannot
  guarantee the global per-IP limit across instances — this is a known, explicitly
  documented limitation, not a blocker for this feature (see Assumptions).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST track, from the rate-limit information Discogs returns
  on each response, how much of the per-minute request budget remains before deciding
  whether to send the next outgoing Discogs request immediately or delay it.
- **FR-002**: The system MUST space out (queue/delay) outgoing Discogs requests once
  the remaining budget in the current window drops below a safety threshold, instead of
  sending them all immediately.
- **FR-003**: The delay added by the local throttle MUST be the minimum needed to stay
  within budget — the system MUST NOT apply a fixed wait unrelated to the actual
  remaining budget.
- **FR-003a**: The throttle MUST apply its delay synchronously to the outgoing Discogs
  request (including when that request is made while handling a live, user-facing
  call such as search or a detail-page lookup), but MUST cap the wait at a short
  maximum; once the cap is reached, the request MUST be sent immediately rather than
  continuing to wait, so a collector-facing request is never held indefinitely.
- **FR-004**: The catalog client and the OAuth collection client MUST share a single
  request-budget tracker, not two independent ones, since both consume the same
  per-IP Discogs limit.
- **FR-005**: When the shared budget-tracking mechanism itself is unavailable or fails,
  the system MUST degrade fail-soft — outgoing requests MUST still be sent (falling
  back to the pre-throttle behavior) rather than being blocked indefinitely or dropped
  because of the throttle's own failure.
- **FR-006**: On cold start, before any real Discogs response has been observed, the
  system MUST assume a conservative default budget based on the documented
  authenticated rate limit, rather than sending requests unthrottled.
- **FR-007**: The existing reactive retry policy and circuit breaker for the catalog
  client MUST continue to apply unchanged; the local throttle is an additional
  preventive layer, not a replacement.
- **FR-008**: The OAuth collection client MUST be covered by the same preventive local
  throttle as the catalog client.
- **FR-009**: When enriching a page of search results with community ratings, the
  system MUST bound the number of Discogs calls in flight at the same time for that
  page to an agreed concurrency limit, instead of firing one call per eligible result
  simultaneously.
- **FR-010**: Search results served entirely from cache (no real Discogs call needed)
  MUST NOT be affected by the new concurrency limit.
- **FR-011**: The existing fail-soft behavior of rating enrichment (a slow or failing
  rating lookup is skipped rather than breaking the search) MUST be preserved exactly
  once the concurrency limit is applied.
- **FR-012**: For safely-retryable collection calls — reads (`listAllInstances`,
  `getFieldMap`) and idempotent mutations (`setRating`, `setFieldValue`,
  `deleteInstance`) — the OAuth collection client MUST retry a 429 or a transient
  5xx/network failure using the same growing-backoff criteria and attempt limit as
  the catalog client, instead of failing on the first attempt.
- **FR-013**: The OAuth collection client MUST continue to fail without retrying on
  non-transient errors (revoked credentials, not-found), with the same error
  translation behavior as today.
- **FR-014**: The catalog client and the OAuth collection client MUST share the same
  circuit-breaker state (contributing to the same exhausted-failure counter), since
  both consume the same per-IP Discogs limit.
- **FR-015**: The system MUST derive its throttling behavior from the actual rate
  limit reported by Discogs on each response rather than a hardcoded value, so it
  adapts if Discogs changes the documented limit.
- **FR-016**: Adding a release to the collection (`addReleaseToCollection`), a
  non-idempotent write, MUST NOT be automatically retried by the local resilience
  layer on a 429 or transient 5xx/network failure; the failure MUST be surfaced to
  the collector exactly as it is today, to avoid creating duplicate collection
  entries.

### Key Entities

- **Rate Limit Budget**: The current, shared understanding of how many Discogs
  requests remain available in the present per-minute window, derived from the
  `X-Discogs-Ratelimit*` response headers and used to decide whether to delay the next
  outgoing request. Shared across the catalog and collection clients.
- **Enrichment Burst**: A group of related Discogs calls triggered by a single
  collector action (e.g. rating lookups for one page of search results, or paging
  through a full library sync) whose in-flight concurrency is bounded.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Collectors stop seeing the "service busy" message more often than the
  reduction already delivered by the existing reactive retry policy, specifically for
  cases caused by the app's own bursts (a search or a library sync) rather than a real
  Discogs outage.
- **SC-002**: A single cold-cache search never triggers more simultaneous Discogs
  calls than the agreed concurrency limit from User Story 2.
- **SC-003**: The rate of outgoing Discogs requests, measured through existing
  operational logs, stays consistently under the documented per-minute limit during
  normal app usage, without relying on a 429 to slow it down after the fact.
- **SC-004**: Library sync and idempotent collection mutations (rating, field edits,
  removing a release) tolerate transient failures to the same degree catalog calls
  already do today; adding a release to the collection continues to surface a
  transient failure immediately, as it does today, to avoid duplicate entries.

## Out of Scope

- Changing the existing reactive retry/circuit-breaker behavior for the catalog
  client (spec `029-discogs-retry-resilience`) — it is reused as-is.
- Coordinating the local throttle across multiple parallel backend instances via a
  distributed store (see Assumptions).
- Changing the error message collectors see when retries are exhausted despite the
  preventive throttle.
- Changing which search results are eligible for rating enrichment (still
  `release`/`master` types, as today).
- Replacing the per-result rating lookup with a batch/bulk endpoint (Discogs does not
  document one) or reducing the default search page size.
- Changing collection mutations' user-facing behavior (they remain synchronous, with
  no queues or manually exposed retry controls in the UI).
- Changing the Discogs OAuth authentication flow itself.
- Fixing the pre-existing e2e test bug referenced in Assumptions — that is tracked
  and resolved separately, outside this feature.

## Assumptions

- This feature does not replace the existing reactive retry-and-circuit-breaker
  policy: it adds a preventive layer (local throttle based on Discogs' response
  headers) on top of it, and extends that reactive layer to the collection client,
  which does not have it today.
- The concrete local-throttle mechanism (in-memory queue, token bucket, shared
  semaphore, etc.), the exact safety threshold on the remaining-budget header at
  which spacing begins, and the exact maximum-wait cap for a synchronously-delayed
  request (FR-003a) are planning-phase decisions, not part of this specification.
- If the project runs multiple instances in parallel, a throttle that only tracks
  state within a single process cannot guarantee the global per-IP limit across
  instances. This feature assumes a best-effort, per-process throttle is already a
  significant improvement over the current state (no preventive throttle at all), and
  that distributed coordination across instances (e.g. via the existing cache store)
  is a possible future evolution, not a requirement here.
- The concurrency limit chosen for User Story 2 is calibrated against the concurrency
  limit already used for library enrichment as a reference point, but the final value
  is a planning-phase decision that may be adjusted based on real-world testing.
- Discogs can change its documented rate limits at any time without notice; the
  throttle is designed to read the real limit reported in response headers on each
  call rather than a fixed hardcoded value, specifically to adapt if the documented
  limit changes.
- During implementation of this feature, the end-to-end (Playwright) test suite MUST
  NOT be run as part of the local development/implementation workflow, because a
  pre-existing e2e bug needs to be fixed first (tracked separately, out of scope for
  this feature). This is a process decision for this implementation cycle only — it
  does not relax the project's standing requirement that frontend-affecting PRs carry
  e2e coverage enforced at the pipeline level once the blocking bug is resolved. This
  feature is primarily a backend change, so e2e coverage exposure is expected to be
  minimal regardless.
