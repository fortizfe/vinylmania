# Quickstart: Validating Application Caching

Prerequisites: this feature is implemented (see `tasks.md` once generated by `/speckit-tasks`). This guide validates the two caching layers end-to-end; it does not include implementation code.

## Prerequisites

- Local Redis instance for backend testing, via the repo's maintained `docker-compose.yml`:
  ```bash
  docker compose up -d redis
  ```
- `backend/.env` (or shell env) with `REDIS_URL=redis://localhost:6379` alongside the existing `DISCOGS_TOKEN`, `FIREBASE_*` vars.
- Both dev servers running: `npm run dev` in `backend/` and `frontend/` (per existing project scripts).

## 1. Backend: verify Discogs response caching (User Story 2)

1. Start the backend with `REDIS_URL` set, and tail its console output (structured JSON logs from `backend/src/config/logger.ts`).
2. Call the search endpoint twice with the same query, e.g.:
   ```bash
   curl -s "http://localhost:3001/api/discogs/search?q=aphex+twin" -H "Authorization: Bearer <token>" > /dev/null
   curl -s "http://localhost:3001/api/discogs/search?q=aphex+twin" -H "Authorization: Bearer <token>" > /dev/null
   ```
3. **Expected**: the first call logs `outcome":"cache_miss"`; the second logs `outcome":"cache_hit"` and the second call's latency is visibly lower. Repeat for `GET /api/discogs/releases/:discogsId`.
4. Stop the local Redis container and repeat step 2. **Expected**: both calls still succeed (200 OK) — logs show `cache_unavailable` instead of an error, confirming FR-008's fallback-to-direct-fetch behavior.

## 2. Frontend: verify instant revisits (User Story 1)

1. Sign in and open the Library list page; open browser devtools Network tab.
2. Open a record's detail page (triggers a network request for that entry).
3. Navigate back to the Library list, then back into the same record detail page.
4. **Expected**: the second visit to that record's detail page renders immediately with no loading skeleton, and the Network tab shows either no new request or a background revalidation request that does not block rendering.

## 3. Frontend + backend: verify no stale data after an edit (User Story 3)

1. On a record's detail page, edit the "Condition" field inline and save.
2. Navigate to the Library list page.
3. **Expected**: the updated condition is visible in the list immediately, without a manual page reload — confirms the mutation hook's cache invalidation (FR-004) is wired correctly.

## 4. Automated checks

- Backend: `npm test` in `backend/` — new tests under `backend/tests/unit/cache/` and updated `backend/tests/contract/discogs.test.ts` must pass.
- Frontend: `npm test` in `frontend/` — new tests under `frontend/tests/unit/queries/` and updated integration tests (e.g. `frontend/tests/integration/recordDetailFlow.test.tsx`) must pass.
- E2E: run the Playwright suite in `e2e/`, including the new `e2e/tests/caching-navigation.spec.ts`, per the constitution's frontend e2e coverage gate.
