# Research: Shared Collapsible Filters with Selectable Lists

## Decision 1: Genre/Style multi-value combined with Discogs' live search the same way Format already is

**Decision**: Reuse the exact existing mechanism `format` already uses against `GET /database/search` — comma-join the selected values into a single string per param (`genre=Rock,Metal`, `style=Grunge,Shoegaze`) and pass them through unchanged to Discogs, via the same `parseFilterParams`/`normalizeFilterValue`/cache-key path already in `backend/src/routes/discogs.ts` and `backend/src/discogs/discogsClient.ts:360-387`. No new backend request-shaping logic is introduced for genre/style; they are promoted from single free-text params to the same shape `format` already has (comma-joined multi-value string).

**Rationale**: `format` already ships a comma-joined multi-value string to Discogs' search endpoint in production (feature 021/022/023). Reusing the identical mechanism for genre/style is the simplest change consistent with Principle III (Simplicity, YAGNI & KISS) and requires no new Discogs integration code — extending a proven request-shaping pattern rather than inventing a second one.

**Alternatives considered**:
- *Issue N parallel Discogs requests (one per selected genre/style value) and merge/dedupe results*: this is the only way to achieve true OR-within-filter matching against live Discogs (see verified behavior below), but was rejected for this feature — meaningfully more complex (merging pagination across N requests, deduplication, higher Discogs request volume against Principle II's "minimize redundant requests"). Revisit only if OR-within-Search-filter becomes an explicit, prioritized requirement in a future feature.

**Verified outcome (post-implementation, live check against `api.discogs.com`)**: comma-joined values are **not** OR-matched by Discogs — `genre=Rock` alone returned 2946 items and `genre=Jazz` alone returned 61 (for the same query), while `genre=Rock,Jazz` returned only 18: far fewer than either individual value, consistent with an AND-like narrowing (only releases satisfying all listed values for that field) rather than a union. This is not a regression introduced by this feature — the exact same AND behavior already existed for `format` before feature 038 (documented in `e2e/tests/search-result-filters.spec.ts`'s "selecting multiple formats narrows results to releases matching all of them together" test, feature 022). **Decision (confirmed with the user)**: accept this as a known, pre-existing platform-level constraint for Search and document it (see spec.md FR-015 and Assumptions) rather than building the parallel-requests-and-merge alternative to force true OR semantics — the added complexity isn't justified for this feature's scope, and the behavior is consistent with what Format has already shipped with since feature 022 without prior complaint. My Library is unaffected: its OR/AND combination is computed in application code (Decision 2) against each entry's own persisted data, independent of Discogs' search API semantics.

## Decision 2: Library filtering strategy — in-app AND/OR filtering over the full library, not a Firestore compound query

**Decision**: When at least one of Genre/Style/Format is active for My Library, the backend fetches the user's **entire** library mirror (reusing the existing `listAllEntries` used today for sync — `backend/src/library/libraryService.ts:83-86`), applies AND-across-filters/OR-within-filter matching against each entry's **persisted** `genre`/`style`/`format` arrays in application code, then paginates the resulting filtered array in memory (slice by `page`/`pageSize`) before enriching only the resulting page's entries for display. When no filter is active, the existing efficient Firestore-side paginated `listEntries` path (`orderBy('addedAt','desc').offset().limit()`) is unchanged.

**Rationale**: Firestore restricts a single query to one `array-contains-any` clause; combining independent OR-within-AND-across conditions on three different array fields (genre, style, format) in one native query is not supported, and building composite indexes for arbitrary subsets of 15×757×51 curated values is infeasible. Per the existing scale assumption for this app (spec 003: "a few hundred records" per user library), fetching the full mirror and filtering/paginating in application code is simple, correct (exact pagination metadata over the true filtered subset, satisfying FR-017), and avoids new indexes — consistent with Principle III. It also requires zero additional Discogs requests, since matching relies entirely on the already-persisted genre/style/format fields (Principle II).

**Alternatives considered**:
- *Firestore composite `array-contains-any` + `in` queries*: rejected — cannot combine three independent array-field OR-conditions in one query, and would need per-combination indexes.
- *Denormalized combined "filter tags" field queried via a single `array-contains-any`*: rejected as unnecessary complexity for the current few-hundred-record scale (YAGNI); revisit only if library sizes grow to a scale where in-memory filtering becomes a measured problem.

## Decision 3: When persisted genre/style/format gets written (backfill mechanism)

**Decision**: Extend the existing per-page enrichment path (`enrichEntries`/`enrichEntry` in `backend/src/library/libraryEnrichment.ts`) so that whenever a release lookup **succeeds** for a library entry, the resolved `genre`/`style`/`format` values are also written back onto that entry's Firestore document (upsert), in addition to being returned for display as they are today. On lookup **failure**, no write occurs — the entry's previously stored values (if any) are left untouched (FR-024, Clarifications Session 2026-07-12).

**Rationale**: This turns every existing enrichment call (which already runs on every library page load, and always on an explicit Refresh via `force=true`) into the backfill mechanism for FR-019, with no new sync machinery, no one-time migration script, and no extra Discogs calls (the release data is already being fetched for display). A brand-new entry that has never been successfully enriched simply has no stored values yet and correctly matches no filter, per the spec's edge case. An explicit Refresh forces a fresh (non-cached) lookup, so it reliably updates stale values per FR-020; a normal cached load's write-back is idempotent and harmless.

**Alternatives considered**:
- *One-time backfill migration script*: rejected — adds an operational step (run once, monitor completion) for no added correctness over the write-back-on-enrichment approach, which self-heals continuously as entries are viewed/refreshed.
- *Only write back on explicit Refresh, not on normal page loads*: rejected — would leave never-refreshed entries permanently stale/unfilterable even though their data is already being fetched anyway for display; strictly worse with no simplicity benefit.

## Decision 4: Static catalog constants generation

**Decision**: Generate `GENRE_OPTIONS` and `STYLE_OPTIONS` frontend constants (mirroring the existing `frontend/src/constants/formatOptions.ts` pattern) once from the supplied `.hu/filters-component-data/genres.json` and `styles.json`, and regenerate `FORMAT_OPTIONS` from `formats-distinct.json` (51 values, replacing the current 33-value list). These are committed as plain `readonly string[]` TypeScript constants, generated/copied once during implementation — no runtime fetch, no build-time codegen pipeline.

**Rationale**: All three catalogs are static reference data (confirmed in spec Assumptions and Edge Cases: no runtime add/remove/reorder support needed) and small enough (largest is 757 short strings, single-digit KB) that a generated static constant is simplest and consistent with the existing `FORMAT_OPTIONS` precedent — no new infrastructure needed (Principle III).

**Alternatives considered**: A build-time script that regenerates the constants from the source JSON/CSV on every build — rejected as unnecessary; the catalogs are fixed reference data, not something expected to change on a recurring basis.

## Decision 5: Collapsible panel + selectable-list component shape

**Decision**: Generalize the existing `FormatFilter` (`frontend/src/components/filters/FormatFilter.tsx`) into a reusable selectable-list filter component (parameterized by label, option list, selected values, and an optional in-list search box), and introduce one new collapsible wrapper component that owns expand/collapse state and the active-filter badge. Both the Search filter form and a new Library filter form compose the same wrapper + the same generalized list component three times (Genre, Style, Format), reusing existing `Card`, `Modal`, `Checkbox`, `Button` atoms exactly as `FormatFilter` does today (mobile: full-screen `Modal`; desktop: an inline/anchored panel using available width, per FR-013).

**Rationale**: `FormatFilter`'s trigger-label logic (`formatTriggerLabel`, comma-list vs `"X (+N)"`), Modal-based mobile presentation, and Checkbox-based multi-select are already exactly the interaction FR-006/FR-007/FR-009/FR-012 ask for — generalizing it (rather than writing three near-duplicate components) satisfies Principle III/IV directly. The in-list search box (FR-008) is the only genuinely new interaction, needed only where the option count makes plain scrolling unusable (Style's 757 values); it's added as an optional prop on the generalized component so Genre (15 values) and Format (51 values) can render without it if a plain list remains usable at those sizes, while Style opts in.

**Alternatives considered**: Three independent components (`GenreFilter`, `StyleFilter`, `FormatFilter`) copy-pasted from the current `FormatFilter` — rejected as a direct violation of Principle III/IV (near-identical logic tripled, one bug fixed in one copy and not the others).
